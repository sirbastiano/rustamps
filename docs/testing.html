Testing

Running tests

python -m pytest -q
python -m build --sdist --wheel
python -m twine check dist/*

The repository includes committed unit tests. Dataset-backed parity tests are kept in-tree as optional checks and skip cleanly when the local validation datasets are absent, so python -m pytest -q remains the clean-checkout gate.

Test structure

  • Default unit tests cover config loading, dataset discovery, acceleration helpers, and Stage-7 helper math.
  • Dataset-backed parity tests and audits require local inputs under inputs_and_outputs/ and are not part of the minimal fresh-clone contract.
  • Parity checks are documented in the verification guide so you can keep the test command list focused.

Coverage

There is no tracked CI workflow or Makefile in this standalone repo snapshot. Validation starts with python -m pytest, python -m build, and python -m twine check, then moves to parity-audit validation in Verification.

In-code assertions and explicit exceptions define many behavior guarantees where test coverage is expected to assert failure conditions.

Validation workflow

Parity audits and explicit verify command patterns are described in Verification.

The verification flow includes contract construction, script orchestration, and CLI comparison commands used by release quality checks.
