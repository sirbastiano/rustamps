Architecture

High-level architecture summary

Execution is split into four layers: command orchestration, pipeline scheduling, stage-portable algorithms, and kernel execution backends.

  • Orchestrator: CLI resolves config and command intent.
  • Scheduler: Stage graph resolves patch and merged workflows.
  • Runtime: Hybrid executor handles IO and CPU parallelism.
  • Numeric kernels: Stage-7 and stage-8 kernels with optional GPU fallback.

Folder structure

PathPurpose
pystamps/cli.pyCommand-line interface and dispatcher
pystamps/config.pyConfig datamodel and YAML/JSON loading
pystamps/pipeline/Stage definitions and ported stage implementations
pystamps/runtime/Hybrid execution primitives
pystamps/io/Dataset and MAT I/O helpers
pystamps/kernels/Kernel registry and backend selector
docs/Generated static documentation site

Core modules and responsibilities

CLI and control

pystamps.cli exposes command parsing and dispatch for status, run, verify, and list-legacy.

Pipeline orchestration

pystamps.pipeline.stages defines stage metadata and dispatch; pystamps.pipeline.ported implements stage behavior.

Execution and backends

pystamps.runtime.executor provides thread/process pools and the task-kind interface. pystamps.kernels chooses CPU/GPU/native implementations via registry.

I/O and verification

pystamps.io.dataset discovers dataset layout; pystamps.io.mat reads/writes MATLAB files; pystamps.verify compares run output with golden datasets.

Data flow

  1. CLI receives command and merges overrides into RunConfig.
  2. Dataset root is discovered into patch list and merged workspace.
  3. Each selected stage checks expected artifacts; if missing it executes implementation or replay mode.
  4. Stage functions produce stage-specific MAT payloads and debug/cached artifacts.
  5. Verification compares produced MAT keys via numeric recursive extraction and tolerances.
If an artifact already exists, the scheduler records skipped_existing and does not re-run the stage.

External integrations

  • Optional GPU backend via cupy (enabled through dependency profile).
  • Legacy helper binaries are not required for all execution paths but may be used in external workflows; discoverable via list-legacy.
  • Compatibility mode supports strict reference replay from another run directory using compat.reference_root.

Design patterns

  • Dataclass-oriented configuration: immutable-ish typed containers for context and results.
  • Command/strategy separation: parsing versus execution separated in cli.
  • Backend registry: named kernel implementations resolved by backend string.
  • Artifact-driven idempotency: stage results are discovered by expected files.
