System map

pySTAMPS is layered by orchestration, stage execution, I/O, and verification.

The architecture is easiest to understand if you separate control flow from numerical work. The CLI decides intent, the pipeline decides stage scheduling, the ported stage functions perform scientific transformations, and verification checks the results against a golden reference.

Core layers

CLI layer

pystamps.cli parses commands and builds the execution context.

Scheduling layer

pystamps.pipeline.stages chooses the stage range, checks existing artifacts, and records results.

Stage layer

pystamps.pipeline.ported contains the concrete stage implementations and many stage-specific helpers.

Support layer

pystamps.io, pystamps.runtime, pystamps.kernels, and pystamps.verify provide the shared infrastructure.

pystamps/verify.pyGolden-dataset comparison and mismatch classification

Data flow

  1. The CLI receives a command such as run, status, or verify.
  2. A RunConfig is loaded, and a PipelineContext is created for pipeline work.
  3. The dataset root is discovered into patch directories and merged-stage artifacts.
  4. Selected stages either reuse existing outputs or execute the ported implementation.
  5. Verification recursively compares numeric payloads in MATLAB files and summarizes failures for debugging.
This artifact-driven design is why pySTAMPS is good at resumable workflows: it can skip work when expected stage outputs are already present.

Design decisions that matter in practice

  • Stage ranges are explicit: you can isolate late stages when debugging or benchmarking.
  • Execution is hybrid: the runtime can use threads for I/O-heavy work and processes for CPU-heavy work.
  • Verification is numeric: comparisons are not text diffs but tolerance-aware checks on MATLAB payloads.
  • Compatibility is configurable: strict reference replay can copy known-good artifacts into a run tree before stage execution.
