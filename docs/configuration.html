Configure a Rustamps run

Rustamps has two deliberately separate configuration layers. A YAML/JSON file controls the runtime and verifier. Scientific processing parameters live in StaMPS-compatible MAT files beside the dataset. Keeping them separate makes it clear whether a value changes scheduling, acceptance criteria, or the scientific method.

The two layers

LayerLocationControls
Run configuration--config run.yamlCPU scheduling, Stage 6 grid/parallelism, and verification tolerances.
Science parametersparms.mat and optional localparms.matFiltering, selection, weeding, unwrapping, reference area, and noise-estimation choices.

Do not put scientific parameters such as unwrap_grid_size in YAML: they are unknown fields there. Likewise, changing a verifier tolerance does not change a pipeline result.

Loading and precedence

  1. Rustamps starts from built-in defaults.
  2. rustamps --config FILE COMMAND overlays the run configuration. A .json suffix selects JSON; every other suffix is read as YAML.
  3. For verify, --profile strict|scientific overrides tolerance.profile.
  4. For run, --cpu-workers N overrides runtime.cpu_workers.

All sections and fields are optional, but unknown names fail immediately. This catches spelling errors instead of silently accepting a configuration that does nothing.

rustamps --config configs/native-kernels.yaml run \
  --dataset /data/INSAR_20200101 --start-step 1 --end-step 8

Minimal explicit YAML

runtime:
  backend: native
  stage2_kernel_backend: native
  stage6_solver: native
  cpu_workers: 0
  stage6_grid_scale: 1.0
  stage6_max_flow_passes: 0
  stage6_ifg_workers: 0

tolerance:
  profile: strict
  rtol: 1.0e-5
  atol: 1.0e-7

You may omit values that equal their defaults. backend, stage2_kernel_backend, and stage6_solver identify the native implementation; they are not performance knobs.

Effective runtime controls

FieldDefaultDetailed behavior
cpu_workers0Zero uses the process-wide Rayon CPU budget. A positive number creates a pool with that many threads. The run-command flag has final precedence.
stage6_grid_scale1.0A finite positive multiplier applied to parms.mat:unwrap_grid_size. Larger values make a coarser unwrap grid: usually faster and less spatially detailed.
stage6_ifg_workers0How many interferograms Stage 6 may solve concurrently. Allowed values are 0 (adaptive), 1, 2, or 4. Adaptive mode respects the CPU and active-grid memory budgets.
stage6_max_flow_passes0Must remain zero, meaning solve integer flow to convergence. Positive pass limits failed scientific validation and are rejected.

Changing stage6_ifg_workers changes scheduling, not the output fingerprint. Changing grid scale changes the scientific sampling and invalidates affected Stage 6 checkpoints.

Native selectors and fixed fields

  • runtime.backend and runtime.stage2_kernel_backend accept native or the alias auto.
  • runtime.stage6_solver accepts native, auto, or backend; all normalize to the in-process native solver.
  • snaphu, external, Python/CUDA providers, and non-empty backend override maps fail closed.

Older schemas still serialize some fields for compatibility, but they have no separate native implementation. If present, these must keep their defaults: io_workers: 8, stage2_native_threads: 0, stage7_chunk_ps: 100000, stage8_chunk_edges: 200000, MAT cache enabled, Stage 2 checkpoint mode final/interval 1, and both debug flags false. Prefer removing them.

The compatibility section must remain at reference_root: null and strict_reference: false. Historical reference replay is a developer workflow, not a production backend.

Verification tolerances

For ordinary numeric values, a comparison passes when:

absolute_error ≤ atol + rtol × absolute(reference)
FieldDefaultMeaning
profilestrictstrict allows no outlier fraction. scientific can allow a reviewed bounded fraction.
rtol / atol1e-5 / 1e-7Global relative and absolute tolerance; both must be finite and non-negative.
wrap_equivalencetrueCompare listed wrapped-phase keys modulo wrap_period.
wrap_period2πFinite positive wrap period.
wrap_keysdph_noiseKeys eligible for wrapped comparison. Unwrapped ph_uw is never wrapped to pass.
exact_keysIdentity/index keysNamed indices, counts, edges, and fingerprints must match exactly regardless of numeric tolerances.
max_outlier_fraction0Fraction in [0, 1]. A nonzero value requires the scientific profile and a hard max_abs.
key_tolerances{}Per-key rtol, atol, outlier fraction, and cap. Unspecified values inherit globals.
 
tolerance:
  profile: scientific
  rtol: 1.0e-5
  atol: 1.0e-7
  key_tolerances:
    ph_uw:
      atol: 0.001
      max_outlier_fraction: 0.001
      max_abs: 0.05

Scientific MAT parameters

parms.mat is normally created or retained with the dataset. For patch stages, PATCH_N/localparms.mat overlays the shared parameters; for merged stages, a dataset-root localparms.mat overlays them. Values are case-insensitive where shown as named choices. Text flags accept y/yes/true/1 and n/no/false/0; a numeric MAT scalar is not a flag encoding.

Checked profiles

  • configs/native-kernels.yaml: strict grid, scale 1.0.
  • configs/stage6-balanced.yaml: scale 4.0 with scientific verification.
  • configs/stage6-fast.yaml: scale 10.0 with reviewed per-key tolerances.
  • configs/stage6-experimental-15x.yaml and 20x: explicitly experimental coarse grids.

A profile is a starting point, not evidence that its tolerance is valid for a new dataset. Compare a representative result against the strict grid before accepting a coarser configuration.
